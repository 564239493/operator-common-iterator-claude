0背景与目标

当前项目的执行链路（executer/runner.py）面向单卡单后端算子：_build_atk_command 硬编码 atk node --backend cpu task ... 或 atk node --backend pyaclnn --devices 0 node --backend cpu task ...， 通过 supports_npu 二元开关切换。这对普通算子足够，但 通算融合算子需要分布式双卡、 双标杆（CPU 标杆 + NPU 级联标杆）+ 精度对比，现有链路无法表达。

用户需求拆解为三步：

  1. 浏览指定算子，发现通算融合算子的可识别特征（先通信后计算 / 命名含 AllToAll·AllGather / 计算融合·并行 / 文档标记“通算融合”）。
  2. 把“通算融合”标识透传到执行阶段，使 case-executor 能据此选用专用执行策略。
  3. 按 4 步执行流程优化当前流程：① CPU 标杆执行 ② NPU 级联标杆执行 ③ 文件改名 ④ 精度对比。
核心约束 用户明确：生成约束和生成用例不需要单独策略（EXTRACT、GENERATE 阶段保持现有逻辑，constraints.json 不改动， 分类元信息写入 run_state.json）；只有 EXECUTE 阶段对通算融合算子启用独立的 4 步流程。
关键既有资产 executer/resources/acc_config.txt 已存在 cv_fused_double_benchmark acc 类型 （含 max_re_ratio / avg_re_ratio / root_mean_squared_ratio 三阈值），generator.py:load_acc_config 已解析，cases.json 的 standard.acc 字段已能承载 per-case 策略 —— 可直接复用为精度对比阈值来源。 但 acc_config.txt 按本方案定义的 7 个通算融合算子仅 aclnnAlltoAllMatmul 一条在列（文件另有 SwinTransformerLnQkvQuant/BatchMatmulWeightNz/ReflectionPad1dBackward 三条 cv_fused_double_benchmark，不属本方案 fusion 范畴），不能作为算子分类的唯一依据，分类必须由 EXTRACT 阶段独立判定。

2.2 Schema 改造点

2.2.1 constraints.json — 不改动（避免影响约束校验）

agent/generators/common_model_definition.py:78-106 OperatorRule 保持 model_config = {"extra": "forbid"}， 不新增任何字段。理由：constraints.json 是约束语义载体，被 validate_constraints 严格校验、被 Z3 生成器直接消费， 混入“算子分类/执行策略”这类元信息会污染约束语义、增加校验复杂度。分类信息改由 orchestrator 派生写入 run_state.json（见 2.2.2）， constraints.json 与用例生成链路完全不受影响。

3.2 路径检查门禁

步骤 1、2 各有一道硬门禁：执行后必须检查对应 dist_*/cases/0/rank_0 与 rank_1 两个独立目录均非空，任一为空即判定任务失败。 这要求 executer/ssh.py 新增远程目录非空检查原语：

async def check_remote_dir_has_files(conn, dir_path: str) -> bool:
    """检查远程目录是否存在且至少含一个文件；对 rank_0、rank_1 分别调用，两者皆须非空。"""
    cmd = f"if [ -d '{dir_path}' ]; then " \
          f"  n=$(find '{dir_path}' -type f 2>/dev/null | head -1 | wc -l); " \
          f"  echo $n; else echo __MISSING__; fi"
    out = (await run(conn, cmd)).strip()
    return out.isdigit() and int(out) > 0
步骤检查路径失败语义
1.1{output_root}/case_{t1}/output/dist_cpu/cases/0/rank_0 与 rank_1任一为空 → status=failed, engine_error="cpu benchmark dist output empty"
2.1{output_root}/case_{t2}/output/dist_npu_bm/cases/0/rank_0 与 rank_1任一为空 → status=failed, engine_error="npu cascaded benchmark dist output empty"
门禁纪律 路径检查失败必须如实写入 engine_error（与现有“网络/认证/环境故障不伪装为 case fail”一致）， 不得降级为普通 case fail 或静默继续步骤 4。步骤 1/2 任一失败即终止 4 步流程，不进入对比。

3.3 与现有 real 三步流程的关系

.claude/skills/execute-cases/SKILL.md 现有 real 模式是 generate → CPU golden 推导 → real-run 三步。 其中“CPU golden 推导”子步骤用 atc-cpu-golden-derivation skill 把 cases_executor.py 的 # TODO: CPU_GOLDEN dummy 块替换为 torch.* 真实计算。

对通算融合算子，CPU 参考由 ATK dist --name cpu 后端自行产出（步骤 1 的 dist_cpu）， 不再需要 torch 侧推导。因此：

real 子步骤default 策略fusion 策略
1. generate（生成 executor + expanded）执行执行（不变，用户要求 GENERATE 不单独策略）
2. CPU golden 推导（atc-cpu-golden-derivation）执行跳过（fusion .tpl 已是真实实现、无 # TODO: CPU_GOLDEN 标记，skill 天然无操作；CPU 标杆由 ATK dist 产出）

executor 生成器（已确认） executer/resources/generator.py:488-604 的 _SPECIAL_TEMPLATES 对 fusion 算子（如 aclnnAlltoAllMatmul） 直接返回专属 .tpl，绕过 j2 dummy 模板。aclnnAlltoAllMatmul.py.tpl 已是完整真实实现：CPU 路径走 manual_all_to_all （dist.send/recv）+ torch.matmul+bias，NPU 标杆走 is_bm 分支的 dist.all_to_all_single，无 dummy、无 # TODO: CPU_GOLDEN 标记。 故 atc-cpu-golden-derivation skill 对 fusion 算子天然无操作（找不到标记即跳过），无需 torch 推导介入，CPU 标杆由 ATK dist --name cpu 后端产出。 “保留 dummy/待确认”假设取消。

3.4 执行层改造点

3.4.1 ATK 命令拼装 — 参数化 phase

executer/runner.py:268-291 当前 _build_atk_command 是单卡单后端、硬编码 --devices 0。 新增 fusion 专用命令构造，按 phase 区分 4 种形态：

def _build_fusion_atk_command(phase, operator_name, remote_paths, devices, num, env_init):
    """phase ∈ {'cpu_benchmark','npu_cascaded','accuracy_load'}"""
    cases_remote = remote_paths.cases_path(operator_name)
    executor_remote = remote_paths.executor_path(operator_name)
    c1, c2 = devices
    common = (f"-c {cases_remote} -p {executor_remote} --save_data output -mt 1 -e {num}")
    if phase == "cpu_benchmark":
        return f"{env_init} && atk node --backend dist --name cpu " \
               f"--devices {c1},{c2} --dist_backend gloo task {common} --task accuracy"
    if phase == "npu_cascaded":
        return f"{env_init} && atk node --backend dist --name npu_bm " \
               f"--devices {c1},{c2} --dist_backend hccl --is_bm true task {common} --task accuracy"
    if phase == "accuracy_load":
        # 见 3.1 步骤4；--output_path 指向步骤2目录
        return f"{env_init} && atk node --backend pyaclnn --is_dist true --task accuracy " \

3.4.3 结果对比与 schema 扩展（精度对比照跑，但不入成败）

成败语义（与 default 的关键差异） default 流程的 passed/failed 来自 ATK --task accuracy 的精度判定（精度超阈=failed，精度定成败）。 fusion 不让精度定成败：用户明确“只需要执行、对比就行，不据精度决定任务成败”。故 fusion 的 passed/failed/total 只反映执行成败：total = N = --num 透传的实际执行用例数（非 atk per-case 精度结果）；三步 atk exit 0 + 路径门禁全真 → passed=N/failed=0；任一执行失败/路径空 → passed=0/failed=N、status=failed。保证 passed+failed==total 恒成立， 精度对比结果独立记入 comparison_result，不影响 status、不影响 passed/failed。

executer/report_parser.py 现有 parse_xlsx_report 解析单份 xlsx（default 复用）。fusion 精度对比解析（仅记录，不产 passed）：

5风险与边界

风险1（已拍板） ATK dist 命令语法按既定写法认定正确。 步骤 4 命令的嵌套 node 子句、--name / --dist_backend / --is_bm / --output_path / accuracy_load 语义与层级按方案 3.1 写定， 落地照写进 runner.py；若执行时发现偏差，后续修正。
风险2 双卡资源与并发。通算融合算子“不支持并发调用、不同的通算融合算子也不支持并发调用” （7 个文档均强调）。iterate-directory 串行迭代目录时，若含多个 fusion 算子必须严格串行； servers.json 的 fusion_devices 不可被同机其他任务占用。
风险3 output_root 路径冲突。步骤 1、2、4 都在 {output_root} 下生成 case_{format-time} 目录。format-time 需保证毫秒级唯一（避免同秒覆盖）； out_t1/out_t2 必须在各自 step 的 atk 执行后立即 find_latest_output_dir 抓取并缓存（见 3.4.2 时序硬约束）， 后续 step 一律用缓存值、不再次 find_latest；步骤 4 的 --output_path 显式指向 out_t2。
边界 不自动回退、不静默 Mock（继承项目既有安全边界）。 fusion_devices 缺失或 supports_fusion 未声明时，fusion 策略直接停止并提示用户补充配置； 路径门禁失败如实写 engine_error，不降级为普通 case fail；不自动提交/推送/删除文件。
边界 GENERATE 阶段不动（用户明确）。通算融合只新增 run_state 执行策略字段与 4 步执行流程， constraints.json schema 不变、约束生成器、用例生成器（Z3 + pairwise）保持现有逻辑；standard.acc 复用已有 fusion acc 阈值（供 step4 对比记录，不入成败），不新增 case 字段。 分类不命中时完全走现有 default 流程，零回归。
